# CloudWatch Monitoring Lab

## 🎯 Lab Goal

This lab focused on understanding how CloudWatch is used to monitor system health and detect abnormal behavior inside AWS environments.

The objective was to learn how monitoring metrics, alarms, and visibility tools help identify operational problems before they become larger failures.

---

## 🧪 Environment Setup

In this lab I configured and tested:

- EC2 monitoring metrics
- CloudWatch alarms for CPU usage
- Instance status check alarms
- Basic log visibility through CloudWatch

The environment was designed to observe how AWS monitoring reacts to changing system conditions.

---

## 🔥 Monitoring Scenarios Tested

I intentionally tested several monitoring situations, including:

- CPU spike simulation
- High resource usage conditions
- Alarm triggering behavior
- Instance health monitoring
- System state changes during stress conditions

These tests helped demonstrate how CloudWatch detects unusual system behavior through metrics and thresholds.

---

## 🔍 Monitoring Workflow

During testing, I practiced:

1. Viewing EC2 performance metrics
2. Creating alarm thresholds
3. Triggering alarms through simulated stress
4. Observing alarm state changes
5. Verifying system health conditions

This helped build a clearer understanding of how monitoring systems provide visibility into infrastructure behavior.

---

## 🧠 Key Technical Learnings

This lab reinforced several important monitoring concepts:

- CloudWatch focuses on system and infrastructure behavior
- Metrics help identify performance and health issues
- Alarms allow automatic detection of abnormal conditions
- Monitoring helps reduce response time during failures
- System health monitoring is different from audit tracking

The lab also improved my understanding of how AWS environments can be monitored proactively instead of reacting only after failures occur.

---

## ⚠️ Real-World Security Insight

Monitoring without alerting is mostly passive visibility.

Without alarms, abnormal conditions can go unnoticed for long periods of time.

This lab focused on understanding how monitoring and alerting improve awareness of:
- performance problems
- system instability
- unusual resource usage
- potential operational abuse
